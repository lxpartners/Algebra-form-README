Sistema de Modos

O formulário opera em dois modos distintos, controlados por um banner sticky abaixo do cabeçalho. O modo é persistido em localStorage e aplicado automaticamente no carregamento.

●
Modo Interno (default)
Todas as secções visíveis e editáveis.
— Secção Referência do Processo: visível e editável, marcada com etiqueta "INTERNO".
— Secção A — Negócio: visível e editável, marcada com "PREENCHIDO PELA ALGEBRA".
— Restante formulário: editável pelo staff interno.
●
Modo Cliente
Vista do cliente — enviada/partilhada com o proponente para preenchimento.
— Secção Referência do Processo: completamente oculta (display:none).
— Secção A — Negócio: visível mas bloqueada (pointer-events:none + overlay), marcada com "PREENCHIDO PELA ALGEBRA".
— Restante formulário: editável pelo cliente.
Implementação Técnica
Variável de estadocurrentModo — 'interno' | 'cliente'
Função toggletoggleModo() → chama applyModo()
PersistêncialocalStorage.setItem('alg_modo', currentModo)
CSS — secção internabody.cliente-mode .internal-section { display:none !important }
CSS — secção readonlybody.cliente-mode .readonly-section — inputs com pointer-events:none + overlay
ValidaçãoCampos da .readonly-section são excluídos de validateRequired() em Modo Cliente
Etiquetas visuais::before pseudo-elemento nas secções com texto de contexto
ℹEm produção, o ideal é gerar um link com parâmetro ?modo=cliente que force o modo automaticamente no carregamento — item no roadmap.
Estrutura do Formulário

Secções do formulário e respectiva visibilidade/editabilidade. Interno apenas staff   Só leitura (cliente) preenchido internamente   Cliente preenchido pelo cliente

—
Referência do Processo Interno
Referência interna gerada automaticamente (ALG-AAAA-XXXX, persistida em localStorage). ID Devedor(es) obrigatório. Identificação do(s) Crédito(s) opcional. Completamente oculta em Modo Cliente.
A
Negócio Só leitura (cliente)
Propósito do negócio, montante da operação, sinal, origem dos fundos, proveniência, meios de pagamento, IBANs. Em Modo Cliente: visível mas todos os campos bloqueados com overlay de protecção e etiqueta "PREENCHIDO PELA ALGEBRA".
B
Proponente Cliente
Toggle Pessoa Singular / Pessoa Coletiva. Singular: dados pessoais, documento (IRN/AIMA/Outra), NIF, residência, profissão, contacto. Secção cônjuge condicional. Coletiva: denominação social, NIPC, CAEs, objeto social, sede, forma jurídica, responsável de contacto, código RCBE. Secção Titulares com blocos adicionáveis dinamicamente.
C
Titulares (Pessoa Coletiva) Cliente
Blocos dinâmicos adicionáveis por titular — nome, cargo, % capital, beneficiário efetivo, NIF, nacionalidade, documento, residência.
D
Representante Legal Cliente
Condicional (Sim/Não). Dados pessoais, documento, NIF, residência, contacto. Todos os campos obrigatórios quando activo.
E
PeP — Pessoa Politicamente Exposta Cliente
4 questões Sim/Não conforme Lei 83/2017 (arts. 2.º, 19.º e 39.º). Ao responder Sim, campos expandem e tornam-se obrigatórios. Definições legais completas em accordion.
F
Documentos Obrigatórios Cliente
Lista distinta por tipo de proponente. Singular: doc. identificação, NIF, procuração, fatura serviço público, recibo vencimento, IBAN. Coletiva: certidão RC + RCBE, docs titulares, NIF, deliberação/procuração, IBAN. Secção comum: comprovativo de origem de fundos (mínimo 1 obrigatório com upload, 6 opções). Formatos: PDF, JPG, PNG. Máx. 10MB por ficheiro.
G
Proteção de Dados & Declarações Cliente
Caixa RGPD em accordion. Declarações do(s) Proponente(s) (4 pontos AML/CFT). Exportação PDF com pré-visualização. Autenticação CMD e upload de assinatura física para 2 proponentes.
H
Anexo I — Definições Legais
Accordion com definições completas: Beneficiário Efetivo (art. 30.º), PeP (13 funções), Membros próximos da família, Pessoas estreitamente associadas, Titular de outro Cargo Político ou Público — tudo conforme Lei n.º 83/2017.

Lógica Condicional

Todos os comportamentos condicionais implementados e a função JS responsável.

Modo Interno → Modo ClientetoggleModo() / applyModo()
Condição: botão no banner clicado
Oculta .internal-section. Bloqueia campos de .readonly-section com overlay. Altera labels do banner. Exclui campos readonly de validateRequired(). Estado guardado em localStorage.
Toggle Singular / ColetivasetType(type)
Condição: botão de tipo clicado
Mostra/oculta #section-singular, #section-coletiva, #docs-singular, #docs-coletiva. Aplica/remove data-req nos campos da secção activa. Salva tipo em localStorage.
Titular — Adicionar / RemoveraddTitular() / removeTitular(n)
Condição: botão "Adicionar Titular" ou "Remover" clicado
Cria/remove bloco HTML com campos de titular. Em modo coletiva activo, aplica data-req automaticamente aos novos campos. Actualiza progresso.
Cônjuge — SimynToggle()
Condição: botão Sim clicado em questão de cônjuge
Secção #conj_det expande. Todos os campos tornam-se obrigatórios. Ao clicar Não: campos limpos e data-req removido.
Representante Legal — SimynToggle()
Condição: botão Sim clicado
Secção #rep_det expande. Todos os campos (exceto NIF estrangeiro) tornam-se obrigatórios.
Entidade Emitente — OutratoggleIssuerOther()
Condição: select = 'outro'
Campo de texto livre aparece para especificar a entidade. Aplicado em Proponente Singular, Representante Legal e Titulares (Coletiva).
PeP — Questões 1 a 4ynToggle()
Condição: botão Sim clicado em qualquer questão PeP
Campos expandem e tornam-se todos obrigatórios, incluindo início de exercício (input[type=month]). Flag PeP activa no PDF e na submissão backend.
Comprovativo de fundos — item marcadockchk click handler
Condição: checkbox de comprovativo marcada
Botão de upload aparece. Se não há ficheiro carregado, mostra aviso. Actualiza validação de documentos. Ao desmarcar: upload e ficheiro limpos.
Proveniência de Fundos / OutrotoggleChkOutro()
Condição: checkbox 'Outro' marcada
Campo de texto livre aparece para especificação. Remove-se ao desmarcar.
RGPD — accordiontoggleGdpr()
Condição: clique no cabeçalho da caixa RGPD
Expande/colapsa o corpo com texto legal completo. Chevron roda 180°.
Definições Legais — accordiontoggleAnnex()
Condição: botão "Ver definições legais" clicado
Expande/colapsa definições legais completas: BE, PeP, membros família, pessoas estreitamente associadas, titular cargo público.
CMD — AutenticaropenCmd(n) / confirmCmd(n)
Condição: botão CMD clicado
Abre portal autenticacao.gov.pt em nova aba. Mostra botões Confirmar/Cancelar. Ao confirmar: marca estado como autenticado, actualiza UI e progresso.
Idioma — PT / ENsetLang(lang)
Condição: botão de idioma clicado
Actualiza todos os elementos data-req-pt/data-req-en e labels do banner de modo. Preferência guardada em localStorage.
Persistência Local

O rascunho do formulário é guardado em localStorage automaticamente.

Chaves localStorage
alg_kycJSON com todos os valores dos campos input, select, textarea (incluindo ref_interna)
alg_kyc_typeTipo de proponente activo: 'singular' | 'coletiva'
alg_modoModo activo: 'interno' | 'cliente'
alg_fontTamanho de fonte: 'normal' | 'large'
⚠Os ficheiros carregados (docFiles) são mantidos apenas em memória durante a sessão — não são persistidos no localStorage por limitação nativa do browser.
ℹA referência interna (ALG-YYYY-XXXX) é gerada uma única vez e guardada em alg_kyc. Nas sessões seguintes é restaurada em vez de regenerada.

Endpoints API

Stack recomendada: Node.js/Express ou Python/FastAPI. Base URL: https://api.algebracapital.pt/api/v1

POST/kyc/submit

Recebe o formulário KYC completo. Valida server-side, guarda na BD, dispara emails automáticos. Suporta Singular e Coletiva.

Body (multipart/form-data)
_typestring'singular' | 'coletiva'
_langstring'pt' | 'en'
_modostring'interno' | 'cliente'
ref_internastringALG-YYYY-XXXX — enviada pelo frontend (localStorage)
pep_flagbooleanTrue se qualquer questão PeP = Sim (calculado no frontend)
[campos]string/dateTodos os campos do formulário — ver secção "Campos do Formulário"
file_*file[]Ficheiros carregados — PDF, JPG, PNG. Máx. 10MB cada.
Response 200
status'ok'
ref'ALG-2026-4823'
idUUID da submissão
created_atISO 8601 timestamp
Response 422
status'error'
code'VALIDATION_FAILED'
fieldsArray de campos em falta ou inválidos
POST/kyc/upload

Upload individual de documento. Retorna URL de armazenamento. Aceite antes ou durante a submissão.

Body (multipart/form-data)
filefilePDF, JPG ou PNG. Máximo 10MB.
doc_typestringSingular: f1–f6 / Coletiva: cc1–cc5 / Fundos: g1–g6 / Assinatura: sign_1, sign_2
ref_internastringPara associar o ficheiro ao processo
Response 200
file_idUUID v4
storage_keyCaminho no S3/Cloudinary
doc_typeChave do documento
GET/kyc/:ref

Detalhe completo de uma submissão. Requer Authorization: Bearer {token}.

Response 200
form_dataObjecto completo com todos os campos
titularesArray de titulares (apenas coletiva)
pep_flagBoolean
documentsArray com metadados e pre-signed URLs (expiração 15min)
status'rascunho' | 'submetido' | 'em_analise' | 'aprovado' | 'rejeitado'
GET/admin/submissions

Lista paginada de submissões. Requer token de admin.

Query Params
pageintPágina (default: 1)
per_pageintPor página (default: 20)
tipostring'singular' | 'coletiva'
pep_flagbooleanFiltrar submissões com PeP = Sim
statusstring'rascunho' | 'submetido' | 'em_analise' | 'aprovado' | 'rejeitado'
POST/cmd/verify

Troca o authorization code CMD por token. Ver secção "Integração CMD" para fluxo completo.

Body (JSON)
codestringAuthorization code recebido do gov.pt
redirect_uristringDeve coincidir com o registado na AMA
proponenteint1 ou 2
Response 200
status'authenticated'
nameNome completo (do id_token)
nifNIF (do id_token)
tokenJWT interno para incluir na submissão

Integração CMD

A autenticação CMD utiliza o protocolo OAuth 2.0 com OpenID Connect disponibilizado pela AMA (Agência para a Modernização Administrativa).

Fluxo de Autenticação
1. Frontend abre popup: https://autenticacao.gov.pt/oauth/authorize ?client_id=<CLIENT_ID> &redirect_uri=https://kyc.algebracapital.pt/cmd/callback &response_type=code &scope=openid%20profile%20nic &state=<state_aleatorio> 2. Utilizador autentica-se no portal gov.pt 3. Gov.pt redirige para o callback com ?code=<code>&state=<state> 4. Frontend envia o code para POST /api/cmd/verify 5. Backend troca o code por access_token: POST https://autenticacao.gov.pt/oauth/token grant_type=authorization_code &code=<code> &redirect_uri=... &client_id=... &client_secret=... 6. Backend valida o id_token (JWT), extrai NIF e nome 7. Backend devolve JWT interno ao frontend 8. Frontend regista autenticação — proponente marcado como autenticado
Estado Actual vs. Produção
Estado actualSimulado — abre portal gov.pt em nova aba e confirma localmente com botão
Para produçãoImplementar fluxo OAuth completo com endpoint /cmd/verify no backend
Registo na AMANecessário registar aplicação em autenticacao.gov.pt para obter CLIENT_ID e CLIENT_SECRET
⚠Guardar CMD_CLIENT_SECRET apenas no servidor — nunca expor no frontend.
